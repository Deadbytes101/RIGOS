THE DEFECTS ARE PART OF THE HISTORY.

RIGOS did not advance by hiding failed assumptions. Physical boot tests exposed authority mistakes that simulation and file inspection could not. Each milestone below records a change in what the machine could actually prove.

CHRONOLOGY

  1. ORIGIN

    Observation was not enough

    The early direction moved from inspecting mining machines toward building a complete bootable appliance. The target became a USB-native system that could own startup, state, runtime generation, mining and recovery locally.

  2. 2026-07-04

    Product contract fixed first

    The project adopted a strict contract: CPU-only, pool-neutral, local-first, no account, no activation, no RIGOS fee, no forced pool and no remote kill switch. Architecture had to preserve that contract rather than patch it in later.

  3. USB PIPELINE

    First bootable image path

    BIOS and UEFI boot paths, immutable appliance roots and a separate persistent-state partition turned the project into an operating image rather than a collection of host scripts.

  4. STATE

    Durable authority separated from root

    Configuration revisions, the current revision pointer, SSH host identity, recovery credential records and health budget moved into persistent state. Runtime files became generated output, not the source of truth.

  5. FIRSTBOOT

    Configuration moved to the physical tty

    The firstboot interface was placed on tty1. Whiptail drawing and answer capture had to use separate streams so the console UI could remain visible while configuration input was committed correctly.

  6. PHYSICAL FAIL

    systemd transaction authority was wrong

    Real hardware showed that a utility-mode conflict could remove firstboot from the initial boot transaction before service conditions were evaluated. The service file looked conditional; the transaction graph had already decided otherwise.

  7. ALPHA.22

    Kernel command line became mode authority

    The fix moved boot-mode authority to kernel command-line conditions and repaired the physical firstboot/systemd transaction defect. Alpha.22 became the first functional physical preview.

  8. ALPHA.23–24

    Hardening after real defects

    Miner health observation, bounded supervision, network inspection, hostname synchronization and a namespace correction for persistent miner-health state were added after physical behavior exposed weak edges.

  9. ALPHA.25

    Operator surface and recorded truth

    The short rig command, public JSON allowlists, automation-friendly status, preserved startup gates and sanitized physical health evidence completed the frozen functional milestone.

  10. CURRENT

    Development paused

    Alpha.25 is frozen as an experimental appliance milestone. It is physically functional, but the project does not claim stable unattended operation or production readiness.

THE IMPORTANT FAILURE

The firstboot defect matters because it demonstrates a low-level systems rule: a service condition does not help when the service has already been removed from the transaction that would evaluate it.

incorrect mental model:
  service enters transaction
      -> ConditionKernelCommandLine evaluated
      -> service skipped or started

physical reality found:
  conflict removes service from transaction
      -> condition never gets authority
      -> firstboot never appears

repair:
  kernel command line defines mode authority
      -> transaction is built correctly
      -> firstboot remains present only in the intended mode

ENGINEERING LESSONS

  • A boot graph is executable architecture. Unit files cannot be reviewed as isolated text.
  • Persistent configuration should be committed and activated, not edited in-place as live runtime state.
  • Recovery must remain observable and explicit; automatic destructive repair destroys evidence.
  • Operator convenience must not bypass the gates that establish state, runtime, huge pages and network truth.
  • Physical samples prove only the machine and conditions actually observed.
